\section{Project Overview}

Lucid Mail is a fully functional email web application built with React and TypeScript,
designed as a direct response to usability shortcomings in existing email clients —
most notably Gmail. Rather than iterating on an existing codebase, Lucid was designed
from first principles: every feature, layout choice, interaction pattern, and visual
decision was validated against a specific, named HCI rule before implementation.

The application provides:

\begin{itemize}[leftmargin=*, itemsep=2pt]
  \item A \textbf{Landing Page} with persuasion-driven onboarding, social proof, and a night-sky
        animation canvas to trigger visceral first impressions.
  \item A \textbf{Top Bar} containing the wordmark, a persistent search field with filters and
        recent-search memory, a zoom widget, settings, and a multi-account switcher.
  \item A \textbf{Resizable Sidebar} that snaps between compact icon mode and expanded text mode,
        housing navigation, custom labels, Meet links, and Chat contacts.
  \item An \textbf{Email List} with a contextual Welcome Header, skeleton loading states, rich
        empty states, and a context-sensitive bulk action bar.
  \item An \textbf{Email Detail} view with a two-column layout: an action sidebar and a main
        content area with smart replies, spam warnings, and context banners for Trash/Archive.
  \item A \textbf{Compose Window} with rich text editing, 30-second autosave, subject-empty
        warnings, recipient autocomplete, file attachments, and Undo Send.
  \item A full \textbf{Settings} page covering theme, density, interface zoom, notifications,
        customisable keyboard shortcuts, tags (custom labels), accounts, and email signature.
  \item A \textbf{Keyboard Shortcuts} system with a discoverable overlay and per-shortcut
        remapping in Settings.
  \item \textbf{Toast Notifications} with inline Undo actions.
  \item A \textbf{Confirm Dialog} for irreversible destructive operations.
\end{itemize}

\section{Design Philosophy}

The guiding philosophy of Lucid is captured in three words the landing page repeats: ``Email,
reimagined for how you think.'' Every decision subordinates visual novelty to cognitive ease.
Tesler's Law — that every system has irreducible complexity — drove us to absorb complexity on
the developer side rather than expose it to users. Zipf's Principle of Least Effort drove us
to make every common action (compose, archive, search) reachable in under two keystrokes.
And Raskin's Three Laws prevented us from ever designing a path that could silently lose a
user's data.

\chapter{Shneiderman's 8 Golden Rules}
% ══════════════════════════════════════════════════════════════

Ben Shneiderman's eight golden rules form the backbone of interaction design evaluation.
This chapter maps each rule to a specific, concrete feature of Lucid Mail.

% ─────────────────────────────────────────────────
\section{S1 — Strive for Consistency}

\begin{principleBox}{S1 · Strive for Consistency}
  Use identical sequences, layouts, terminology, and UI actions across every
  part of the system. Same error style, same button positions, same terminology.
\end{principleBox}

Consistency in Lucid operates at four levels:

\textbf{Visual vocabulary.} Every destructive action (Delete, Delete forever, Sign Out)
uses the same \textcolor{danger}{red} colour and the same icon family (Feather icons).
Every positive confirmation uses \textcolor{success}{green}. This rule is applied in the
Bulk Action Bar, the Email Detail action sidebar, the Accounts Settings tab, and the
Tags Settings tab — four separate components sharing one visual language.

\textbf{Button hierarchy.} Primary calls-to-action (Send, Create tag, Sign in) always use
the \texttt{btn-primary} class — blue background, white text, 7px/18px padding. Secondary
actions always use \texttt{btn-outline} — transparent background, border, muted text.
Destructive confirm buttons always inherit the danger colour. No exception was permitted.

\textbf{Terminology.} The application never calls the same thing by two names. The
navigation label ``Archive'' is identical to the toast message ``Archived'', the bulk bar
button text ``Archive'', and the keyboard shortcut description ``Archive selected.'' Users
who learn the term in one location recognise it in all others.

\textbf{Keyboard shortcut hints.} Everywhere a shortcut exists, it is shown in
parentheses in the tooltip — \texttt{Compose ( C )}, \texttt{Search mail ( / )},
\texttt{Keyboard shortcuts ( ? )}. The formatting pattern is always the same.

\section{S2 — Enable Frequent Users to Use Shortcuts}

\begin{principleBox}{S2 · Enable Shortcuts}
  Advanced users should bypass steps that slow them down;
  novices still follow the full path. Keyboard shortcuts, macros, autocomplete.
\end{principleBox}

Lucid implements a fully layered shortcut system, arguably the most thorough in any
student-built email client.

\textbf{Single-key shortcuts:} \texttt{C} (compose), \texttt{E} (archive selected),
\texttt{\#} (delete selected), \texttt{Z} (undo last action), \texttt{/} (focus search),
\texttt{?} (open shortcuts overlay).

\textbf{Two-key sequence shortcuts:} \texttt{G I} (go to Inbox), \texttt{G S} (go to Sent),
\texttt{G T} (go to Trash), \texttt{G D} (go to Drafts). These are debounced within 800 ms,
matching Gmail's own two-key navigation model (Jakob's Law, L8).

\textbf{Compose window shortcuts:} \texttt{Ctrl+B} (bold), \texttt{Ctrl+I} (italic),
\texttt{Ctrl+U} (underline), \texttt{Ctrl+Enter} (send). These mirror standard word-processor
conventions so existing muscle memory transfers immediately.

\textbf{Discovery without recall.} The \texttt{?} key opens a full overlay that lists every
shortcut, organised into three categories: Actions, Navigation, Search \& Help. The overlay
is also reachable from the Help icon in the Top Bar — satisfying both the expert (hotkey) and
the novice (visual discovery) paths simultaneously.

\textbf{Remappable shortcuts.} In Settings → Shortcuts, each shortcut has a ``badge'' that
can be clicked to capture a new key press. This is the only email client in this class that
implements per-shortcut remapping — directly supporting the Power Law of Practice (L5), because
users who have already built muscle memory on different clients can retrain Lucid to match.

\section{S3 — Offer Informative Feedback}

\begin{principleBox}{S3 · Informative Feedback}
  Every action should have a visible, proportional system response.
  Minor actions: small feedback. Major actions: substantial feedback.
\end{principleBox}

Lucid applies proportional feedback across four tiers:

\textbf{Tier 1 — Micro feedback (hover/focus states).} Every button changes opacity or
background on hover. Every text input changes border colour to \texttt{brand-500} on focus
and returns to \texttt{border} on blur. This is implemented identically in the search bar,
compose recipient inputs, subject input, tag name input, and Settings text areas.

\textbf{Tier 2 — Status indicators.} The Compose Window title bar shows a live
\texttt{Saved} / \texttt{Saving…} indicator using a small \textcolor{success}{green}
checkmark, giving the user continuous awareness that their draft is safe — without interrupting
the writing flow. This responds to Raskin's First Law: never allow work to be lost.

\textbf{Tier 3 — Toast notifications.} Every significant action (archive, trash, send,
restore, permanently delete) produces a toast at the bottom of the screen with the action's
result in plain English: ``Archived'', ``Moved to Trash'', ``Message sent''. Toasts for
reversible actions include an ``Undo'' link. Toasts for irreversible ones (``Permanently
deleted'') carry no undo link — deliberately, because offering false hope of reversal
would violate user trust.

\textbf{Tier 4 — Loading skeletons.} When the email list is loading, eight skeleton rows
are rendered — animated grey bars matching the shape of real email rows. This uses the
Gestalt Closure principle (G1): the brain reads the familiar shape and understands ``data is
coming'' without any text instruction.

\section{S4 — Design Dialogs to Yield Closure}

\begin{principleBox}{S4 · Dialogs for Closure}
  Multi-step tasks should have a clear, recognisable end state.
  Users should never wonder ``did it work?''
\end{principleBox}

Closure is implemented in Lucid through \textit{context banners} — full-width inline strips
that appear at the top of the Email Detail view when the email's current state requires user
awareness.

When an email in the \textbf{Trash} is opened, a warm amber banner reads:
\textit{``This email is in Trash. Messages are permanently deleted after 30 days.''}
Two action buttons — Restore to Inbox and Delete forever — are directly embedded. The user
instantly knows the email's fate, their remaining options, and the time window.

When an email in the \textbf{Archive} is opened, a blue banner reads:
\textit{``This email is archived and not in your Inbox.''} This directly answers the
question users frequently ask in Gmail: ``where did my email go?''

After \textbf{sending}, the compose window closes and a success toast appears: ``Message
sent'' — confirming the action without requiring the user to navigate to Sent to verify.

After \textbf{login}, the inbox appears immediately and the Welcome Header greets the user
by first name with live stats — confirming successful session establishment without a
separate ``You are now logged in'' screen.

\section{S5 — Permit Easy Reversal of Actions}

\begin{principleBox}{S5 · Permit Reversal}
  Users should be able to undo any reversible action.
  Reduces anxiety and encourages exploration.
\end{principleBox}

Reversal in Lucid is implemented at three levels, each matching the severity of the action:

\textbf{Level 1 — Instant undo via toast.} Archive, Trash, Move, and Mark Read operations
display a toast with an ``Undo'' button. Clicking it calls \texttt{undoLastAction()} from the
email store, which applies a stored \texttt{UndoAction} snapshot to restore the previous state.

\textbf{Level 2 — Undo keyboard shortcut.} The \texttt{Z} key triggers the same undo
mechanism from the keyboard without requiring the user to reach for the mouse. If there is
nothing to undo, a toast reads ``Nothing to undo'' — preventing a silent no-op.

\textbf{Level 3 — Soft-delete.} Deleting an email moves it to Trash, not permanent
deletion. The Trash auto-purges after 30 days. Users have three weeks to discover and recover
a misdeleted email. This matches Gmail's own model (Jakob's Law) but is made explicit with
the Trash banner (S4).

\textbf{Destructive confirmation.} Permanently deleting from Trash requires explicit
confirmation via a \texttt{ConfirmDialog} with the label ``Delete forever (cannot be undone)''.
The destructive button is rendered in red. The cancel button is the visually dominant path,
implementing the Fitts' Law principle of making the dangerous target smaller and less prominent.

\section{S6 — Error Prevention and Handling}

\begin{principleBox}{S6 · Error Prevention}
  Prevent errors first by design; when unavoidable, provide plain-language
  recovery guidance.
\end{principleBox}

\textbf{Prevention — subject field guard.} If the user attempts to send a message with an
empty subject line, the send is intercepted. The subject field's border turns
\textcolor{danger}{red} and a warning bar appears: \textit{``Subject is empty.''} alongside
a ``Send anyway'' escape hatch — respecting autonomy (S7) while preventing the most common
composing mistake.

\textbf{Prevention — send button disablement.} The Send button is disabled when the
\texttt{To} field is empty. Users cannot send an email with no recipient, regardless of
compose window state.

\textbf{Prevention — tag name uniqueness.} In Settings → Tags, creating a duplicate tag
name triggers an immediate inline error: \textit{``A tag with this name already exists.''}
before the form is submitted — constraining the interaction to valid inputs only (Norman's
Constraints, D3).

\textbf{Prevention — email validation at login.} The login modal validates format
(\texttt{@} required) before submission and marks the field in red with
\textit{``Enter a valid email address.''} The error attaches to the specific field rather than
appearing at the top of the form — placing the error where the user's attention already is
(Raskin's Third Law).

\textbf{Recovery — compose autosave.} Every 30 seconds of inactivity, the compose window
silently saves a draft. If the browser closes or the window crashes, the draft is recoverable
from the Drafts folder — directly implementing Raskin's First Law.

\section{S7 — Support Internal Locus of Control}

\begin{principleBox}{S7 · Locus of Control}
  The user always feels they are controlling the system, not being controlled by it.
  No auto-popups, no surprise navigation, no system-initiated changes.
\end{principleBox}

\textbf{Resizable sidebar.} The sidebar has a drag handle on its right edge. Users can pull
it to any width between 68px (compact icon mode) and 300px. At 120px it snaps to either
compact or expanded — providing predictable snap behaviour while still letting the user control
the breakpoint they want. No setting menu is required; the affordance is the handle itself.

\textbf{Settings are immediate.} In the Settings page, there is no ``Save'' button.
Every toggle, segmented control, or option card applies its change instantly. The subtitle
``Changes are saved automatically'' appears below the Signature textarea to make this explicit.
Users never have to wonder whether their preferences were committed.

\textbf{No auto-submission.} The Compose Window does not send on tab-out, on window-blur,
or on any system event. Only the Send button or \texttt{Ctrl+Enter} triggers sending. The
close button (\texttt{×}) autosaves the draft and closes — it never sends.

\textbf{Compose window controls.} The title bar exposes Minimise, Fullscreen, and Close
buttons. The window can be minimised to a 48px strip (to keep context while reading), or
expanded to a full-screen editor for long messages. The user decides the modality, not the
system.

\textbf{Account switching.} The account switcher in the Top Bar shows all signed-in
accounts. The user decides which one is active at any moment. The system never auto-switches
based on an incoming email from a different account.

\section{S8 — Reduce Short-Term Memory Load}

\begin{principleBox}{S8 · Reduce STM Load}
  Don't make users remember information across screens. Display what they need,
  when they need it. Related to Miller's 7$\pm$2 Law.
\end{principleBox}

\textbf{Welcome Header.} When the inbox is open, a Welcome Header above the email list
displays the active account's name, avatar, and three live statistics: Unread count,
Sent count, and Drafts count. Users never need to navigate to Sent or Drafts just to
find out how many they have.

\textbf{Unread badge in sidebar.} When the sidebar is in expanded mode, unread counts
appear as coloured badges on each nav item. In compact mode, an unread dot-badge
(\texttt{unread-dot-badge}) signals the presence of unread mail without revealing the
number — respecting the compact layout while preserving the signal.

\textbf{Recent searches.} The search dropdown surfaces the last five search queries
when focused with an empty field. Users do not need to remember what they previously
searched — the system remembers and offers recognition over recall.

\textbf{Context-aware bulk bar.} When emails are selected, the bulk action bar shows
only actions appropriate for the current folder. In Trash, it shows ``Restore'' and
``Delete forever'' — not ``Archive'' (which would be nonsensical). In Archive, it shows
``Move to Inbox'' and ``Move to Trash.'' The relevant memory of ``what folder am I in''
is embedded in the available actions, not required from the user.

\textbf{Email detail — sender and date always visible.} The sender's name, email, and
full timestamp are always displayed in the Email Detail header, regardless of scroll position.
Users who receive emails a week later do not need to remember when they arrived.

% ─────────────────────────────────────────────────
\section{N1 — Visibility of System Status}

The system must always communicate its current state via timely, appropriate feedback.

In Lucid, the active navigation item receives \texttt{aria-current="page"} and is
highlighted with a blue background (\texttt{brand-50}) and bold text — making the current
location unambiguous at all times. The search bar placeholder text includes \texttt{( / )} to
signal the keyboard shortcut, surfacing system capability as status information.

The Compose Window title bar serves as a persistent status strip: it shows the email's
subject (or ``New message''), a save-status indicator (\texttt{\faCheck\ Saved} / \texttt{Saving…}),
and the window-state buttons. Even when minimised to 48px, the subject is visible — the user
always knows what message they are working on.


\begin{figure}[H]
  \centering

\section{N2 — Match Between System and Real World}

Terminology and metaphors must come from the user's world, not technical internals.

Lucid avoids all internal jargon. The folders are named Inbox, Starred, Sent, Archive,
Drafts, Spam, and Trash — the same nouns English-speaking users have used in email since
the 1990s (Jakob's Law). Icons reinforce the metaphors: a trash can for Trash, an envelope
for Inbox, a star for Starred, a paper aeroplane for Send.

The compose body placeholder reads: ``Write your message… (Ctrl+B bold, Ctrl+I italic,
Ctrl+U underline)'' — plain English with inline teaching, not ``ContentEditable region
with rich text support enabled.''

Error messages use the same register: ``Enter a valid email address'' (not ``Invalid
input format''), ``Subject is empty'' (not ``Missing required header field''), ``No results
for X'' with a suggestion to ``try different keywords'' (not ``Query returned 0 records'').

\section{N3 — User Control and Freedom}

Users often take wrong turns; always provide a clear exit.

Every view in Lucid has an escape. The Email Detail has a ``Back'' button with a left arrow.
The Settings page has a back button in its header. Compose windows have a close (×) button,
a minimise button, and the keyboard \texttt{Escape} key dismisses the shortcuts overlay.
The bulk-select mode (selecting emails by checkbox) has an explicit close button (×) in the
bulk bar, returning to normal mode. No modal or mode exists in Lucid without a visible,
labelled escape.

The search bar's clear button (\texttt{×}) appears as soon as text is typed, removing the
need to manually select-all and delete — a one-click escape from the search state.

\section{N4 — Consistency and Standards}

Users should not wonder whether different words mean the same thing.

Platform conventions are followed rigorously. The top bar layout — logo left, search
centre, account right — mirrors Google's own application chrome (Jakob's Law). Dropdown
menus all share the \texttt{.dropdown} class: same border, same border-radius, same shadow,
same item hover style. Every dropdown fades in with a \texttt{fade-up} animation at the same
speed (150ms).

Icon semantics are locked: the same star icon means starred/unstarred in the email list,
the email detail sidebar, and the bulk action bar. The same trash icon means delete in
compose (discard draft), email rows (delete), and the detail sidebar (move to trash).

\section{N5 — Error Prevention}

Design proactively eliminates error-prone situations.

Beyond the subject-guard and recipient-guard described under S6, Lucid prevents errors through
\textbf{constraint} (Norman D3). The Send button is a \texttt{disabled} HTML attribute —
not just visually dimmed — when the To field is empty, which means it is truly unreachable
by keyboard focus.

In the Tags tab, the ``Create tag'' button is the only submission path; pressing Enter
in the name field triggers the same handler, preventing accidental blank-name tags.

In the zoom widget, the \texttt{-} button is \texttt{disabled} at 70\% and the \texttt{+}
button is \texttt{disabled} at 150\%, preventing the interface from ever reaching an
unreadable state through repeated clicks.

\section{N6 — Recognition over Recall}

Show options; do not make users remember commands.

The sidebar labels display icons alongside text when expanded. In compact mode, icons alone
are shown — but every compact item has a \texttt{Tooltip} revealing its full label on hover,
bridging recognition and recall. Custom label buttons in compact mode show the label's first
letter with its unique colour — recognition of the colour replaces recall of the name.

The recipient autocomplete in the Compose Window surfaces a dropdown of matching contacts
as the user types. Users do not need to remember full email addresses — the first two
characters trigger recognition-based selection.

The search dropdown shows recent searches when the field is focused and empty — surfacing
history without requiring the user to remember what they previously typed.

\section{N7 — Flexibility and Efficiency of Use}

Cater to both novice and expert users.

Lucid explicitly supports two usage modes. \textbf{Novice users} navigate by clicking the
sidebar, reading tooltip labels, using the visible search bar, and clicking action buttons.
\textbf{Expert users} compose with \texttt{C}, search with \texttt{/}, archive with
\texttt{E}, navigate with \texttt{G I / G S}, and format with \texttt{Ctrl+B}.

The density setting (Compact / Default / Spacious) lets expert users who process email
quickly choose Compact — seeing more emails per screen — while less frequent users prefer
Spacious for better legibility.

The zoom widget in the Top Bar lets users with high-resolution monitors scale down to 70\%
for information density, while users with accessibility needs scale up to 150\%.

% ─────────────────────────────────────────────────
\section{N8 — Aesthetic and Minimalist Design}

Every element competes for attention. Irrelevant information reduces the visibility of important information.

Lucid's inbox view contains exactly four pieces of information per email row:
the sender avatar (identity), the subject and preview (content), the timestamp (recency),
and a star (importance). No advertisement banners, no promotional score widgets, no
``Promotions'' / ``Social'' / ``Updates'' tab noise from Gmail.

The Email Detail's action sidebar is a single column of icon-only buttons with no labels —
relying on \texttt{Tooltip} for discovery. This keeps the main reading area at full width
without a distracting toolbar row across the top.

The Settings page uses generous whitespace (40px horizontal, 48px vertical padding) and
groups each setting into a clearly titled section — consistent with Simon's Law: visual
hierarchy reduces perceived complexity.

% ─────────────────────────────────────────────────
\section{N9 — Help Users Recover from Errors}

Error messages must identify what went wrong and how to fix it.

Every error in Lucid is inline, co-located with the field that caused it, written in
first-person-familiar English. In the login modal: \textit{``Enter a valid email address''}
appears directly below the email field in \textcolor{danger}{red}. In the Tags tab:
\textit{``A tag with this name already exists.''} appears below the name input.
In the compose warning bar: \textit{``Subject is empty.''} appears immediately below the subject
row with a ``Send anyway'' escape — acknowledging the error while respecting that the user
might have a legitimate reason to send without a subject.

None of these messages reference internal states, codes, or HTTP methods.

The shortcut descriptions are written as verbs (``Compose new email'', ``Archive selected'',

Tooltips are the primary in-context help mechanism: every icon-only button in the sidebar,
the Top Bar, and the Email Detail sidebar has a \texttt{Tooltip} with a concise action

\chapter{Norman's 6 Design Principles}
% ══════════════════════════════════════════════════════════════

Donald Norman's principles, introduced in \textit{The Design of Everyday Things},
focus on the psychology of interaction at the individual control level.

\section{D1 — Visibility}

Functions must be visible to be discoverable. Hidden equals unused.

The Compose button is the topmost element in the sidebar in both compact and expanded modes.
It is the first thing the sidebar renders, matching the Primacy Effect (L2). In compact mode
it is a 38px circle with a \texttt{+} — unmissable at any sidebar width. In expanded mode it
becomes a full-width blue pill with the word ``Compose.''

The search bar spans the entire centre of the Top Bar, with \texttt{flex: 1; max-width: 720px}.
It is always visible; it is never behind a click-to-reveal icon. This directly addresses
Gmail's mobile regression where the search field is collapsed to an icon.


\begin{figure}[H]
  \centering
  \includegraphics[width=\linewidth]{expand_closure.jpg}
  \caption{Compose button in compact (circle) and expanded (pill) sidebar modes}
\end{figure}

\section{D2 — Feedback}

Every action must produce an immediate, perceivable response.

Beyond the toast and save-status systems already described, Lucid adds micro-feedback at the
input level. The tag colour selector in Settings uses a circular button grid; the selected
colour gets a 3px dark border \textit{and} a box-shadow ring in the same colour — two
simultaneous feedback signals to accommodate both high-contrast and colour-dependent users
(Universal Design U4).

The Login button shows a spinning CSS animation (\texttt{animation: spin .7s linear infinite})
during the authentication flow — communicating ``the system is working'' without blocking
the UI or losing the user's context.

\section{D3 — Constraints}

Restrict interaction to valid actions only.

Physical constraints: the sidebar resize handle can only be dragged horizontally; the minimum
width (68px) and maximum width (300px) are enforced at the JavaScript event level.

Logical constraints: the zoom widget buttons are disabled at the boundary values. The Send
button is genuinely disabled (not just styled) when the To field is empty. The tag ``Create''
button allows submission only when the name field is non-empty.

Cultural constraints: the Delete forever button is styled in red because red universally
signals danger in Western UI conventions. The Restore button is blue (brand action) because
blue signals trusted, safe navigation.

\section{D4 — Mapping}

Controls must logically relate to their effects.

The sidebar resize handle appears on the \textit{right edge} of the sidebar — dragging
rightward makes it wider, dragging leftward makes it narrower. The direction of the drag maps
directly to the direction of the effect, matching the steering-wheel analogy (same direction).

Compose window window-control buttons appear in the \textit{title bar} of the compose window —
not in the body or the footer. This maps to the OS convention (Windows, macOS) where window
controls live at the top-right of window chrome.

The Cc and Bcc toggles in the Compose Window appear at the right edge of the To/From area —
the logical location where ``additional recipients'' would be added in a physical letter-writing
metaphor.

\section{D5 — Consistency}

Similar interface elements do similar things. Rules have no exceptions.

All left-click interactions select or activate. All hover states use the same 120ms
transition. All disabled states use \texttt{opacity: 0.5}. All icon buttons (\texttt{icon-btn})
are 36$\times$36px squares with \texttt{border-radius: var(--radius-sm)} and the same
hover background: \texttt{var(--gray-100)}.

\section{D6 — Affordance}

Object appearance communicates how it should be used.

The Compose circle button (\texttt{+}) affords clicking with a clear circular shape and a
recognisable plus symbol. The sidebar resize handle is a 4px-wide strip that changes cursor
to \texttt{col-resize} on hover — affording horizontal dragging. Every text input field has
a 1.5px border that changes colour on focus — affording text entry. The star icon affords
toggling (filled = active, outline = inactive).

The zoom widget's \texttt{-} and \texttt{+} labels directly signal their arithmetic effects
— minus decreases, plus increases — without requiring any label or documentation.

\end{figure}

\chapter{Gestalt Principles}
% ══════════════════════════════════════════════════════════════

The Gestalt school of perceptual psychology provides the theoretical basis for how users
group, separate, and navigate visual information in Lucid's interface.

\section{G1 — Closure (Reification)}

The brain fills in missing information to complete familiar shapes. Lucid uses skeleton
loading rows — animated grey bars in the exact proportions of a real email row (avatar circle,
sender bar, preview bar, timestamp bar) — so that users perceive incoming data before it
arrives. The shape is recognised; the content is mentally anticipated. This prevents the
disorienting ``blank page flash'' that triggers negative visceral reactions (P3).

\section{G2 — Common Region}

Elements inside a bordered area are perceived as one group.

The bulk action bar (\texttt{bulk-bar}) is a visually distinct strip with its own background
colour (\texttt{var(--brand-50)}), top border (\texttt{1px solid var(--brand-100)}), and
height. When it appears, it is immediately perceived as a region separate from the email list
below — a ``control panel'' for the selection. Users do not confuse it with the email header.

Each email row is bounded by a subtle bottom border, which creates micro-regions that group
each email's metadata (avatar, sender, subject, preview, timestamp) into one scannable unit.

\section{G3 — Proximity}

Elements close together are perceived as related.

In the Email Detail view, the two-column layout uses physical separation as a grouping
mechanism: the action sidebar (star, archive, trash, spam) is a narrow left column, visually
close together, perceived as one group of ``actions on this email.'' The main content
(subject, sender, body, smart replies, reply/forward buttons) fills the right column —
separated by a visible border.

In the Settings page, each setting has its label and description on the left, and its
control (toggle, segmented control, option cards) on the right — maintaining proximity of
label to control throughout all six tabs.


\begin{figure}[H]
  \centering
  \includegraphics[width=\linewidth]{spam.jpg}
  \caption{Email Detail two-column layout: action sidebar (left) and content (right)}
\end{figure}

\section{G4 — Common Fate}

Elements moving together are seen as one group.

In the search dropdown, all results slide in together with the same \texttt{fade-up}
animation. The result group is perceived as a unified response to the search query,
not as individual independent items appearing sequentially.

When a compose window is minimised, the title bar collapses smoothly — the entire window
shrinks as one element, reinforcing that it is a single object changing state, not a set of
disparate controls being hidden.

\section{G5 — Symmetry}

Symmetrical compositions are perceived as stable.

The landing page hero section is centred both horizontally and vertically within the viewport.
The wordmark, headline, subheadline, CTAs, and fine print are all centre-aligned — producing
a symmetrical composition that conveys calm authority. The feature cards grid and testimonials
grid both use \texttt{auto-fit} with equal-width columns — symmetric layout signals order
and credibility.

The Compose Window is consistently right-aligned at a fixed \texttt{right: 16px} offset —
multiple windows offset by a fixed 556px — so that even with multiple compose windows open,
the layout remains geometrically predictable.

\section{G6 — Similarity}

Elements sharing visual traits are perceived as related.

All navigation items in the sidebar share the same height (40px), the same icon size
(16$\times$16px), the same font size (0.875rem), and the same hover background. They are
immediately perceived as a navigation group.

All toast notifications share the same bottom-fixed position, the same border-radius, the
same shadow, and the same typographic hierarchy — sender name bold, message body regular,
Undo link in brand colour. A user who sees one toast knows how to read every subsequent one.

\section{G7 — Figure and Ground}

The brain separates foreground objects from background.

When a compose window is open, a semi-transparent dark overlay (\texttt{rgba(32,33,36,.28)})
with a \texttt{backdropFilter: blur(3px)} is applied to the entire application behind it.
This blurs the background into the ``ground'' and brings the compose window to the
``figure'' — making the modal's priority unambiguous without completely hiding the
context behind it.

The same technique is used for the login modal on the landing page (\texttt{rgba(0,0,0,.6)}
with \texttt{blur(8px)}) — the page remains visible but subordinate.


\begin{figure}[H]
  \centering
  \includegraphics[width=\linewidth]{2.jpg}
  \caption{Email Detail two-column layout: action sidebar (left) and content (right)}
\end{figure}

\section{G8 — Continuity}

Elements aligned on a smooth path or line are seen as connected.

The Settings page left navigation is a vertical list of six tab buttons — all left-edge
aligned, all with identical spacing (2px gap). The eye travels smoothly down the list
perceiving it as a single navigation rail, not six independent controls.

In the Email Detail, the subject, sender section, and body are all left-aligned, creating a
vertical reading line that the eye follows naturally top-to-bottom without interruption.

\section{L1 — Miller's Law (7$\pm$2)}

Short-term memory holds approximately seven chunks of information.

The primary sidebar navigation is divided: only four items (Inbox, Starred, Sent, Archive)
are shown by default. A ``More folders'' toggle reveals the remaining four (All Mail, Drafts,
Spam, Trash). This 4+4 chunking keeps both layers within Miller's limit. The Settings page
has exactly six tabs — within the 7$\pm$2 range. The Shortcuts overlay has three categories,
each with four to six shortcuts.

\section{L2 — Primacy and Recency Effect}

Users best remember the first and last items in a list.

In the sidebar, Inbox is always first — the most-used destination. In the login modal,
``Email \& Password'' is the first tab — the most common authentication method. In the
Settings tabs, ``Appearance'' is first (the most exploratory users try) and ``Signature''
is last (the setting users return to finish their setup). In the landing page navigation,
``Sign in'' is last (recency) and ``Get started free'' is the final call-to-action.

\section{L3 — Fitts' Law}

Click time increases with distance and decreases with target size.

The Compose button is the largest clickable element in the sidebar in expanded mode —
a full-width 44px-high pill — because Compose is the most frequently initiated action.
In compact mode it is a 38$\times$38px circle, meeting Apple's 44$\times$44px minimum
touch target recommendation.

The permanent-delete confirmation button in the ConfirmDialog is smaller and bordered
in red — a deliberately smaller, visually less prominent target to reduce accidental
permanent deletion.

The zoom \texttt{-} and \texttt{+} buttons are 24px squares — small but adjacent to
each other for quick fine-tuning. The spacing between them and the label is minimal,
reducing the distance users must travel between presses.

\section{L4 — Hick's Law}

Decision time increases with the number of choices.

Lucid reduces choices at every decision point. The Compose Window's formatting toolbar
has exactly seven actions (Bold, Italic, Underline, Link, Bullet list, Blockquote, Attach)
— the primary formatting operations only, not a full word-processor toolbar.

The search filters surface only two options: ``Has attachment'' (toggle) and ``From:''
(text). Advanced filter logic is not exposed at the UI level — complexity is absorbed
(Tesler's Law, L9).

The Settings tabs are labelled with single-concept nouns (Appearance, Notifications,
Shortcuts, Tags, Accounts, Signature). No tab combines two concepts, preventing the
``where would I find this?'' choice paralysis.

\section{L5 — Power Law of Practice}

Task completion time decreases as a function of practice trials.

Lucid's keyboard shortcut system is designed to accelerate with practice. The first time
a user opens the app, they navigate via mouse. After ten sessions, the \texttt{C} key for
compose becomes automatic. The two-key sequences (\texttt{G I}, \texttt{G S}) feel
difficult on day one but are near-instant after a week — matching the logarithmic learning
curve.

The shortcut remap feature directly acknowledges that users coming from other clients
(\textit{e.g.} Superhuman: \texttt{E} for archive) have already built practice-curve
memory that should not be erased. Remapping \texttt{E} or \texttt{\#} costs zero
retraining.

\section{L6 — Pareto Principle (80/20)}

80\% of usage comes from 20\% of features.

The primary 20\% in an email client: compose, read, search, archive, delete. Every one
of these is accessible in Lucid with a single input: \texttt{C}, click-to-read, \texttt{/},
\texttt{E}, \texttt{\#}. The remaining 80\% of features (tags, notifications, signature,
meeting links, chat) are progressively disclosed through the sidebar ``More'' toggle and
the Settings page — visible and reachable, but not competing for attention on the main
interface.

\section{L7 — Zipf's Principle of Least Effort}

Users want maximum outcome for minimum effort.

\textbf{Smart replies} are the clearest manifestation: after reading an email, the user
sees three one-click replies pre-generated from the email context. A single click opens
a compose window pre-filled with the response — reducing a 30-second task to 2 seconds.

\textbf{Recipient autocomplete} reduces email address entry from a full typing task to a
two-character recognition-and-select operation.

\textbf{Bulk operations} allow selecting all emails in a label with one checkbox toggle,
then archiving or deleting all with one button — compressing N repetitive actions into one.

\section{L8 — Jakob's Law of Usability}

Users expect your site to work like every other site they use.

Lucid follows the standard web application shell:
\begin{itemize}[leftmargin=*, itemsep=2pt]
  \item Logo top-left → returns to inbox on click.
  \item Search bar centred in the top bar.
  \item Account/profile controls at top-right.
  \item Navigation in the left sidebar.
  \item Content in the main right panel.
  \item Footer on the landing page: Privacy, Terms, Security, Help.
\end{itemize}

The LUCID wordmark uses the exact Google brand colours (Blue, Red, Yellow, Green, Blue)
to leverage existing colour associations with trusted, familiar software. Users who have
used Gmail for a decade will read this as a deliberate reference — reducing the novelty
anxiety of switching clients.


\section{L9 — Tesler's Law (Conservation of Complexity)}

Every application has irreducible complexity. If removed from the UI, it moves to the developer.

The autocomplete logic in the Compose Window silently parses the To/Cc/Bcc fields, splits
on commas, isolates the last entry, and filters contacts — absorbing the complexity of
email-address parsing. The user types a name fragment and selects.

The search system applies filters from two sources (the visual filter chips and the inline
From: field) and merges them with the query string in a \texttt{useSearch} hook — the user
toggles a chip and results update; all intersection logic is invisible.

Draft autosave reconstructs a full email object (with sender, recipients, labels, thread ID)
from partial compose state every 30 seconds — the user never configures this, it just works.

\section{L10 — Raskin's 3 Laws}

\textbf{Law 1 — Never harm your work.} Autosave every 30 seconds. Soft-delete with a
30-day recovery window. Undo for all reversible actions. Permanent delete requires a
two-step confirmation.

\textbf{Law 2 — Never waste your time.} Recipient autocomplete. Smart replies. Bulk
actions. Keyboard shortcuts. The system infers the draft save ID (\texttt{draft-\{id\}})
rather than asking the user to name their drafts.

\textbf{Law 3 — Be humane.} Error messages are co-located with the field that caused them.
The subject-empty warning appears immediately below the subject row — where the user is
already looking — not at the top of the compose window where it would be outside their
locus of attention. The Caps Lock indicator example from the reference document inspired
placing all validation inline.

All elements feel harmonious and belong together.

\chapter{Universal Design Principles}
% ══════════════════════════════════════════════════════════════

Universal Design's seven principles ensure that Lucid works for users regardless of
ability, experience, language proficiency, or physical context.

\section{U1 — Equitable Use}

Every interactive element in Lucid carries both a visible label or icon and an
\texttt{aria-label} attribute for screen readers. The search input has
\texttt{role="searchbox"} and \texttt{aria-label="Search mail"}. The sidebar nav has
\texttt{role="navigation"} and \texttt{aria-label="Mail navigation"}. The email list has
\texttt{role="list"} and \texttt{aria-label="Email list"}. The compose window has
\texttt{role="dialog"} and \texttt{aria-label="Compose email"}.

No feature is available exclusively to mouse users. Every action has a keyboard pathway:
keyboard shortcuts, Tab-accessible buttons, Enter-submittable forms.

\section{U2 — Flexibility in Use}

\textcolor{red}{\textbf{Lucid accommodates right-handed, left-handed, and keyboard-only users. The sidebar canbe resized from either direction. The zoom widget scales the entire interface, including all touch targets proportionally.}}Three keyboard navigation paradigms are supported
simultaneously: single-key shortcuts, two-key navigation sequences, and modifier-key
combinations in the compose editor.

\section{U3 — Simple and Intuitive Use}

\textcolor{red}{\textbf{The login modal eliminates complexity for first-time users: three tabs (Email, Google,
GitHub) with clear labels}, a loading indicator during authentication, and a ``Continue
as Guest (Demo)'' fallback that requires zero credentials.}

Empty states across all folders use plain language: ``Your inbox is empty. When you receive
mail, it will appear here.'' There are no error codes, no ``null result set'' messages,
and no technical state descriptions exposed to the user.

\section{U4 — Perceptible Information}

\textbf{Visual + textual:} All icon buttons pair with tooltip text. Star icons are filled vs.
outlined — two visual channels for starred state (shape and fill), not just colour.

\textbf{Colour blindness:} No information is conveyed by colour alone. Unread emails are
bold (typographic channel) and \texttt{var(--text-primary)} colour (contrast channel).
Danger actions are red and labelled (``Delete forever'', ``Sign out''). The tag colour
selector uses circular buttons with border contrast rings — the selected state is indicated
by a 3px border regardless of colour.

\textbf{Motor accessibility:} All interactive elements are keyboard-reachable. The sidebar
resize handle is mouse-only, but all its effects (switching folders, reaching navigation)
have keyboard alternatives.

\section{U5 — Tolerance for Error}

\textcolor{red}{\textbf{The 30-day Trash window is the primary error-tolerance mechanism}: accidental deletion is
recoverable for four weeks. The autosave draft system ensures no compose-window content
is ever permanently lost. The ``Send anyway'' button on the empty-subject warning respects
user intent while preventing accidental sends without a subject.}

\begin{figure}[H]
  \centering
  \includegraphics[width=\linewidth]{reversal.jpg}
  \caption{Tolerance for error}
\end{figure}

\section{U6 — Low Physical Effort}

The zoom widget reduces the physical scanning distance for users who need a larger interface.
At 150\% zoom, all hit targets scale proportionally, reducing the motor precision required
for accurate clicking. Keyboard shortcuts eliminate mouse travel entirely for power users,
directly implementing this principle for expert users.

\begin{figure}[H]
  \centering
  \includegraphics[width=\linewidth]{resize.jpg}
  \caption{The zoom widget reduces the physical scanning distance for users who need a larger interface}
\end{figure}

\section{U7 — Size and Space}

All primary interactive targets meet or exceed the 44$\times$44px Apple Human Interface
Guidelines minimum. Sidebar navigation items are 40px tall. The Compose button is 38px in
diameter (compact) or 44px in expanded mode. The Account Pill button is 36px tall. The zoom
widget's \texttt{-}/\texttt{+} buttons are 24$\times$24px — below the 44px ideal, but
grouped together within a larger 80px interactive zone that encompasses both and the label.

\section{P1 — Aesthetic Usability Effect}

Attractive designs are perceived as more usable, even if they aren't.

The landing page leads with the most visually ambitious element in the project: a
\texttt{<canvas>}-based animated night sky with 280 twinkle-animated stars that respond
to cursor proximity with a radial blue glow. This is not functional — it does not send
email. But it establishes a visceral first impression (P3) of craftsmanship and care that
directly increases the perceived usability and trustworthiness of the entire application.

The LUCID wordmark in Google colours, the frosted-glass login modal with
\texttt{border-radius: 24px} and \texttt{box-shadow: 0 32px 80px rgba(0,0,0,.25)}, and the
browser chrome mockup in the hero section all signal: this product is finished, polished,
and professional.

\section{P2 — Von Restorff Effect (Isolation Effect)}

Distinctive items are remembered better. One different element stands out.

Lucid's landing page has a single primary CTA — ``Get started free'' in brand blue.
Everything around it is either dark background (hero) or white card (feature section) or
dark testimonial cards. The blue button is the only blue element on the page — it cannot
be missed.

In the inbox, the only coloured element on a typical email row is the avatar (a coloured
circle unique to each sender). All rows share the same white background, same grey text,
same layout — so when one row is blue-highlighted (currently previewed email), it is
immediately distinctive.

Users form instant emotional judgments about design.

The night sky landing page is specifically designed to trigger a ``wow'' reaction within
the first 100ms of the page load — before the user reads a single word. The star proximity
interaction (stars brightening when the cursor is near) rewards mouse movement with a
haptic-like visual response, creating an immediate sense of a living, reactive interface.

\section{P4 — Dual-Coding Theory}

Words and images together produce stronger memory encoding than either alone.

Each feature card on the landing page pairs an emoji icon (visual channel) with a title and
description (verbal channel): \texttt{\faBolt\ Blazing fast}, \texttt{\faMoon\ Night \& day},
\texttt{\faReply\ Undo anything}. This emoji+text pairing is not decorative — it creates a
dual-coded memory trace: users who forget the title may still remember the lightning bolt.

Smart reply buttons (``Sounds great!'', ``Thanks for letting me know.'') pair an
implicit tone signal (the choice of words) with the visual affordance of a pill button — the
visual form signals ``this is a complete, ready-to-use response,'' the words signal the

\section{Issue 1 — Gmail's Tab System Buries Important Emails}

\begin{gmailIssueBox}{Gmail Issue: Promotions / Social / Updates Tabs}
  Gmail's five-tab inbox (Primary, Social, Promotions, Updates, Forums) fragments the
  user's attention and routinely buries important emails in non-Primary tabs. Users
  frequently miss time-sensitive messages.
\end{gmailIssueBox}

\begin{lucidSolutionBox}{Lucid Solution: Single Unified Inbox with Custom Tags}
  Lucid presents a single Inbox with all messages in chronological order. Users who
  want categorisation create \textbf{custom Tags} (Settings → Tags) with their own
  colours and names — not a pre-imposed, opaque categorisation algorithm.
\end{lucidSolutionBox}

\textbf{Principle citations:} S7 (Locus of Control — user controls organisation),
N7 (Flexibility — power users create tags; casual users don't),
L9 (Tesler's Law — categorisation complexity moved to user choice, not system imposition).

\section{Issue 2 — Gmail's Cognitive Load from Feature Density}

\begin{gmailIssueBox}{Gmail Issue: Overwhelming Feature Surface Area}
  Gmail exposes Meet links, Chat, Tasks, Calendar events, Google Workspace integrations,
  Smart Compose suggestions, and multiple toolbars simultaneously — creating a high
  cognitive load for users who simply want to read and write email.
\end{gmailIssueBox}

\begin{lucidSolutionBox}{Lucid Solution: Progressive Disclosure by Section}
  The sidebar divides features into clearly labelled sections (Menu / Meet / Chat) with
  a section divider and label. The four primary nav items (Inbox, Starred, Sent, Archive)
  are always visible; secondary items (Drafts, Spam, Trash, All Mail) are behind a
  ``More folders'' toggle.
\end{lucidSolutionBox}

\textbf{Principle citations:} L4 (Hick's Law — fewer simultaneous choices),
N8 (Aesthetic Minimalism — every element competes for attention),
W6 (Simplicity — KISS principle).

\section{Issue 13 — Gmail's Reload Gives No Feedback or Closure}

\begin{gmailIssueBox}{Gmail Issue: Silent Reload — No Status Signal}
  Gmail's reload button (the circular-arrow icon next to the select-all checkbox)
  produces no loading indicator on click. On a first press it sometimes shows a
  one-frame spinner, but gives no closure message on completion. Subsequent presses
  produce nothing at all. Users are left uncertain whether the reload occurred,
  whether any new mail arrived, and whether the system is still working.
  This is a textbook violation of Nielsen's Heuristic 1 (Visibility of System Status).
\end{gmailIssueBox}

\begin{lucidSolutionBox}{Lucid Solution: Three-Phase Reload with Full Closure}
  Lucid's reload is designed with three distinct HCI-grounded phases:
  \begin{enumerate}[leftmargin=*, itemsep=2pt]
    \item \textbf{Immediate icon feedback (S3):} The \texttt{IconRefresh} circular-arrow
          icon begins spinning at 0.75s/revolution the moment the button is clicked,
          signalling that work has begun — zero ambiguity about whether the press registered.
    \item \textbf{Skeleton loading rows (G1 Closure):} The email list is replaced by
          eight animated skeleton rows in the exact proportions of real email rows
          (avatar circle, sender bar, preview bar, timestamp bar). The brain perceives
          the incoming content before it arrives, eliminating the disorienting blank-page
          flash.
    \item \textbf{Closure toast on completion (S4):} When the reload finishes, a
          contextual toast fires — \textit{``Inbox up to date · N unread''} if unread
          messages exist, or \textit{``All caught up — no new messages''} otherwise.
          This is a specific, factual outcome statement, not a generic ``Done.'' The user
          knows the exact result without looking anywhere else.
  \end{enumerate}
  A 3-second cooldown (N5 Error Prevention) prevents accidental double-fires that
  would flood the interface with overlapping skeleton states. The button uses
  \texttt{aria-label} that dynamically updates to ``Checking for new mail\ldots''
  during the reload — supporting screen-reader users (U1 Equitable Use).
\end{lucidSolutionBox}

\textbf{Principle citations:} S3 (Informative Feedback — spin icon + ``Checking'' toast),
S4 (Closure — outcome toast after reload completes),
G1 (Closure/Reification — skeleton rows anticipate real content),
N1 (Visibility of System Status — every phase is visible),
N5 (Error Prevention — 3-second cooldown),
D6 (Affordance — circular-arrow universally signals reload),
L3 (Fitts' Law — button adjacent to label in header's F-pattern zone),
U1 (Equitable Use — dynamic aria-label for screen readers).

\section{Issue 14 — Gmail's Stats Header Is Purely Decorative}

\begin{gmailIssueBox}{Gmail Issue: Non-Interactive Status Information}
  Gmail provides no at-a-glance overview of account state (unread count, today's
  sent count, draft count) without manual folder navigation. Even if such counts
  were displayed, clicking them in Gmail performs no action — information is
  presented but not made actionable. This creates unnecessary cognitive load:
  users who want to jump to unread mail must navigate to Inbox and scroll,
  and those who want to check today's sent mail must click the Sent folder and
  mentally filter by date.
\end{gmailIssueBox}

\begin{lucidSolutionBox}{Lucid Solution: Actionable Stat Badges with Instant Drill-Down}
  Lucid's Welcome Header displays three interactive stat badges, each triggering
  a context-appropriate action:

  \begin{itemize}[leftmargin=*, itemsep=4pt]
    \item \textbf{Unread badge} — Shows the current unread count in brand blue.
          Clicking it applies an \textit{unread-only filter} to the inbox without
          navigating away from it. The badge turns blue with a border (N1 Visibility)
          and the label changes to ``Unread \texttimes'' to signal the active state.
          A dismissable chip (``Unread only ×'') appears in the list header as an
          always-visible escape hatch (S7 Locus of Control, N3 User Control).
          Clicking the badge again, clicking the chip, or navigating away all
          revert the filter — matching users' expected reversibility (S5).

    \item \textbf{Sent badge} — Shows only \textit{today's} sent count, not a
          cumulative all-time figure. This matches the user's mental model
          (N2 Match Real World — ``How many did I send today?'' not ``How many
          have I ever sent?''). Clicking navigates to the Sent folder pre-filtered
          to today, with the list header reading ``Sent · Today (today only)''.
          A ``Today only ×'' chip clears the filter to reveal all sent mail.

    \item \textbf{Drafts badge} — Shows the current draft count. Clicking navigates
          directly to the Drafts folder — one click from the inbox instead of two
          (sidebar click).
  \end{itemize}
\end{lucidSolutionBox}

\textbf{Principle citations:} S7 (Locus of Control — user activates and reverts
filters at will), N1 (Visibility — active filter state reflected in badge colour
and chip), N3 (User Control and Freedom — escape hatch always one click away),
N2 (Match Real World — Sent badge shows today's count, not lifetime total),
N7 (Flexibility and Efficiency — one-click drill-down vs. two-click sidebar navigation),
S8 (Reduce STM Load — no need to remember folder names or navigate to find counts),
L7 (Zipf's Least Effort — maximum information with minimum clicks).

\section{Context-Sensitive Help (Tooltips)}

Every icon-only button in Lucid (sidebar compact mode, Top Bar icon buttons, Email Detail
action sidebar, Compose toolbar) has a \texttt{Tooltip} component that appears on hover
without disrupting the workflow. The tooltip content is written as an action description,
not a feature name: ``Compose new email'' rather than ``Compose''; ``Archive (removes from
inbox)'' rather than ``Archive''; ``Delete — moves to Trash (30 days to recover)'' rather
than ``Delete''. This satisfies User Support requirements \textbf{SP1 (Availability)},
\textbf{SP2 (Accuracy)}, and \textbf{SP3 (Unobtrusiveness)}.

The \texttt{?} key and the Help icon (Top Bar) open a full-screen overlay categorising all

Lucid uses semantic HTML roles throughout:

Custom tag colours in compact sidebar mode are supplemented by the first letter of the

\end{table}

\section{Final Reflection}

Lucid Mail demonstrates that the gap between a feature-rich product like Gmail and a
\textit{usable} product is not one of capability, but of design intent. Gmail has every
feature Lucid has, and hundreds more. Yet by adding features without principled constraints,
Gmail has accumulated cognitive debt: a Promotions tab that buries real emails, a fixed
sidebar that cannot accommodate power users, a compose window whose size is non-negotiable,
and no at-a-glance dashboard for the user's own account state.

Lucid's approach — one principle, one feature, every time — produces an interface that
is objectively narrower in feature count but subjectively easier to use. Tesler's Law
reminds us that the complexity did not disappear; it was relocated from the user's working
memory to the developer's implementation. That is the correct trade-off for a well-designed
human-centred system.

The 40 years of HCI research encoded in Shneiderman, Nielsen, Norman, Gestalt, Raskin,
Krug, and their contemporaries are not academic exercises. They are empirical findings
about how human cognition works under load. Lucid is an argument in code that
following them systematically produces a product that users describe not as ``simple''
or ``limited'' but as ``clean'', ``fast'', and — most importantly — ``it just works.''

\chapter{Loading States \& Onboarding}
\addcontentsline{toc}{chapter}{Loading States \& Onboarding}
% ══════════════════════════════════════════════════════════════

Loading and first-run experiences are among the most overlooked surfaces in email client
design, yet they set the user's entire mental model of a product. Lucid implements three
distinct loading treatments, each grounded in a specific HCI principle.

% ── 12.1 ───────────────────────────────────────────────────────
\section{Login Loading Screen — Animated Wordmark}

\begin{tcolorbox}[enhanced, breakable, colback=brandlight, colframe=principleblue, left=6pt, right=6pt, top=5pt, bottom=5pt, boxrule=1.2pt, arc=4pt]
  \textbf{Principle anchor:} Shneiderman S3 (Informative Feedback) · Gestalt G1 (Closure)
  · Aesthetic Usability Effect P1
\end{tcolorbox}

Immediately after the user authenticates, Lucid displays a full-screen interstitial
featuring the ``LUCID'' wordmark rendered in a neutral grey. A left-to-right colour sweep
progressively fills each letter with its signature Google colour
(\textcolor{brand}{\#4285F4} · \textcolor{danger}{\#EA4335} · \textcolor{warning}{\#FBBC04} · \textcolor{success}{\#34A853})
as a rainbow progress bar beneath it advances in parallel.

\textbf{Design rationale.}
\begin{itemize}
  \item \textit{Informative Feedback (S3):} The advancing colour fill gives the user
        continuous, proportional feedback about progress — not a spinner that communicates
        nothing about how long remains.
  \item \textit{Closure (G1):} Partial colouring creates a perceptual ``gap'' the brain
        actively wants to close. Users perceive the incomplete state as temporary rather than
        broken, reducing impatience.
  \item \textit{Aesthetic Usability Effect (P1):} The on-brand animation leaves a premium
        first impression that colours the user's evaluation of every subsequent interaction
        (Aesthetic Usability Effect, Kurosu \& Kashimura, 1995).
  \item \textit{Familiar pattern — novel execution:} Progress bars are a known affordance
        (Jakob's Law, L8). Threading them through the product wordmark is distinctive without
        being disorienting.
\end{itemize}

The screen fades out over 480 ms once the animation completes, transitioning smoothly to
the skeleton or inbox without a jarring cut.

  \centering
  \includegraphics[width=\linewidth]{loading.jpg}
  \caption{Login Loading Screen: LUCID wordmark colour sweep with rainbow progress bar beneath}
\end{figure}

% ── 12.2 ───────────────────────────────────────────────────────
\section{Inbox Skeleton — Full-Layout Shimmer}

\begin{principleBox}{Principle Anchor}
  \textbf{Principle anchor:} Gestalt G1 (Closure) · Shneiderman S3 (Feedback) ·
  Norman N1 (Visibility of System Status)
\end{principleBox}

When the main application mounts after login, Lucid renders a pixel-accurate skeleton of
the complete three-column layout — topbar, sidebar navigation, and a column of twelve email
rows — for 1.3 seconds before the real data appears. Every skeleton element pulses at
1.6 s using a \texttt{sk-pulse} CSS keyframe animation that oscillates opacity between 1
and 0.45.

\textbf{Why a full-layout skeleton outperforms a spinner:}
\begin{enumerate}
  \item \textbf{Spatial priming.} The skeleton occupies exactly the space the real content
        will fill. Users' eyes land in the right place before data arrives, eliminating the
        jarring reflow that occurs when a spinner is replaced by a full layout (Luke
        Wroblewski, ``Mobile First'', 2011).
  \item \textbf{Perceived performance.} Skeleton screens consistently score 20–30\% faster
        on perceived-load tests than equivalent spinner-based loading (Facebook, 2013;
        Lincoln Loop, 2015). Lucid's 1.3 s real-data delay is masked almost entirely by the
        skeleton's active appearance.
  \item \textbf{Closure (G1).} Partially rendered shapes trigger the Gestalt law of closure:
        the brain interprets incomplete rectangles as email rows and completes the picture,
        mentally ``pre-loading'' the inbox before it arrives.
  \item \textbf{Structural hints.} The skeleton sidebar reproduces the item count and active
        highlight of the real sidebar, communicating the app's information architecture
        before a single byte of email data has loaded.
\end{enumerate}

The skeleton uses the shared \texttt{.skeleton} CSS class (background \texttt{var(--gray-100)},
border-radius \texttt{var(--radius-sm)}) for visual consistency with the in-list loading
rows already present in the email list component.

  \centering
  \includegraphics[width=\linewidth]{inbox.jpg}
  \caption{Inbox Skeleton: pixel-accurate three-column layout (topbar, sidebar, email rows) pulsing before real data arrives}
\end{figure}

% ── 12.3 ───────────────────────────────────────────────────────
\section{Onboarding Tutorial — Blurred Backdrop Popup}

\begin{principleBox}{Principle Anchor}
  \textbf{Principle anchor:} Jakob's Law L8 · Shneiderman S8 (Reduce STM Load) ·
  Cialdini P2 (Liking) · Norman N9 (Help \& Documentation)
\end{principleBox}

First-time users — identified by the absence of a \texttt{lucid\_onboarded\_v1}
\texttt{localStorage} key — are presented with a five-slide modal tutorial immediately after
the skeleton resolves. The tutorial appears over a 10 px Gaussian-blur backdrop
(\texttt{backdrop-filter: blur(10px)}), keeping the inbox visible but defocused so the
user retains spatial context while reading the tutorial.

\subsection*{Slide structure}

\begin{center}
\begin{tabular}{@{}lll@{}}
  \toprule
  \textbf{Slide} & \textbf{Title} & \textbf{Core message} \\
  \midrule
  1 & Welcome to Lucid        & Side-by-side comparison: what's the same vs.\ what's better \\
  2 & Familiar layout         & Three-column layout identical to Gmail — zero learning curve \\
  3 & Keyboard shortcuts      & C / E / Z / ? — all the power-user keys at a glance \\
  4 & Undo anything, anytime  & Contrast with Gmail's 30-second undo-send limitation \\
  5 & You're all set          & Confirmation + hint to press ? for the full shortcut overlay \\
  \bottomrule
\end{tabular}
\end{center}

Each slide pairs a custom SVG illustration with concise text. Inline SVG was chosen over
raster images to guarantee pixel-perfect rendering at all display densities without
network round trips.

\textbf{Design rationale.}
\begin{itemize}
  \item \textit{Jakob's Law (L8):} Framing Lucid relative to Gmail explicitly transfers the
        user's existing mental model. Slide 1 lists every familiar affordance alongside the
        specific improvement Lucid makes — reducing re-learning to zero.
  \item \textit{Reduce STM Load (S8):} Five short slides present one idea each. Chunking
        (Miller's Law) keeps working memory within its 7 ± 2 capacity.
  \item \textit{Blurred backdrop — figure/ground (G3/G7):} The defocused inbox behind the
        modal establishes the modal as figure and the inbox as ground, creating clear visual
        hierarchy without hiding the spatial context of where the user has arrived.
  \item \textit{Liking (P2):} The illustrated, conversational tone of the tutorial increases
        affinity with the product before the user has written a single email (Cialdini's
        Liking principle — people respond positively to things that feel personal and warm).
  \item \textit{Skip affordance:} A ``Skip'' button is always visible in the top-right corner
        of the modal, honouring user autonomy (N3 User Control) and avoiding the dark pattern
        of forced onboarding flows.
  \item \textit{Progressive dots + Back navigation:} Dot indicators (current slide
        highlighted in brand blue, others in \texttt{var(--gray-200)}) communicate position
        within the sequence without requiring the user to count slides. Back navigation
        supports exploration rather than forcing a linear path.
  \item \textit{One-time display:} The flag is stored in \texttt{localStorage} rather than
        a session variable, so returning users are never shown the tutorial again — respecting
        their expertise and eliminating the patronising repetition that plagues many web apps.
\end{itemize}

% ── 12.4 ───────────────────────────────────────────────────────
\section{Loading State Sequence Summary}

The three loading surfaces form a deliberate \textbf{temporal funnel}:

\begin{center}
\begin{tabular}{@{}clll@{}}
  \toprule
  \textbf{Phase} & \textbf{Surface} & \textbf{Duration} & \textbf{User state} \\
  \midrule
  1 & Login loading screen (wordmark fill) & $\approx$2.2 s & Authenticating     \\
  2 & Inbox skeleton (full layout shimmer) & $\approx$1.3 s & App bootstrapping  \\
  3 & Onboarding tutorial (blurred modal)  & User-controlled & First-run only     \\
  \bottomrule
\end{tabular}
\end{center}

Each phase has a distinct visual language — brand wordmark animation, structural skeleton,
contextual modal — so the user always knows \textit{why} they are waiting and \textit{what}
comes next. Together they transform an unavoidable delay into a designed moment that builds
brand familiarity, transfers mental models, and sets expectations before the user sends
their first email.
